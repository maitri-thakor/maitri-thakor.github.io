1.Were blocked on a simple problem:

I was working on a coding problem that required me to iterate through an array of numbers and apply a certain rule to each number. The rule was that if a number was divisible by 3, I had to replace it with the string "Fizz". If a number was divisible by 5, I had to replace it with the string "Buzz". If a number was divisible by both 3 and 5, I had to replace it with the string "FizzBuzz".

The problem I was facing was that my code was not working as expected. It was replacing all numbers with "FizzBuzz", even though some numbers were not divisible by 3 or 5.

To solve this problem, I used a different approach. Instead of using a single function to check for divisibility by 3, 5, and both 3 and 5, I created a separate function for each condition. This made my code easier to understand and debug.

I felt a mix of frustration and satisfaction throughout the process. Frustration because I couldn't figure out why my original code wasn't working as expected. Satisfaction because I was able to find a solution by breaking down the problem into smaller parts and using a different approach.

What I learned from this experience is that sometimes, the best way to solve a problem is to break it down into smaller, more manageable parts. This allows you to focus on one part of the problem at a time, making it easier to identify and fix any issues.

2. Elegantly solved a problem:

The problem was to complete the getAverage function that takes a name parameter and returns the average of that student's test scores. The gradebook object was provided, and we could call the average function within the getAverage function.

To solve this problem elegantly, I read the problem statement carefully and made sure I understood what was being asked.I knew that I needed to access the testScores property of the student object in the gradebook object. I also knew that I could use the average function to calculate the average of the scores.

I used the studentName parameter to access the student object in the gradebook object. I then accessed the testScores property of the student object and passed it to the average function.

I felt confident and focused throughout the process. I knew what I needed to do and how to do it. I was able to write the solution quickly and easily for this code.

This problem was a good reminder that sometimes the most elegant solution is the simplest one. By focusing on the problem and breaking it down into smaller parts, I was able to write a solution that was concise and easy to understand.

Trying something: This approach helps me identify problems and fix them quickly. It also allows me to experiment with different solutions and see what works best.
Rubber ducky method: I have used the rubber ducky method a few times, and I find it helpful.
Reading error messages : I am very confident in my ability to read and understand error messages.sometimes error messages can be hard to understand, which can be frustrating.
Console.logging : Console.logging allows me to see the state of my variables and functions at different points in the code. It's a quick and easy way to check if my code is doing what I expect it to do.
Googling : I am very confident in my ability to find answers to programming questions through Google. It's very difficult when i have short of time.
Asking your peers for help : I have a supportive facilitator who are always willing to lend a hand.

Improving your process with reflection : I feel very confident in my ability to reflect on my problem-solving process and make changes as needed.
